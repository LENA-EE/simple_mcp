## 1. Назначение и область действия

Система ДРОСПР / JARVIS — AI-ассистент для понимания и рефакторинга легаси Perl-кода внутри периметра банка.

**Что система делает сейчас (MVP):**
- Статический анализ Perl-кода через Perl::Critic
- Возврат структурированного JSON с найденными проблемами
- LLM на стороне клиента (Kilo / Cline / opencode) объясняет проблемы разработчику
- Блокировка плохого git push через pre-push hook

**Что система НЕ делает сейчас и НЕ планирует без явного RFC:**
- Не изменяет код автоматически — никогда, ни при каких условиях
- Не отправляет код за пределы периметра банка
- Не имеет прямого доступа к репозиторию из LLM-компонентов
- Не принимает решений без подтверждения человека

## 2. Абсолютные ограничения (HARD CONSTRAINTS)

> Нарушение любого из этих пунктов — основание для немедленного останова системы.
> Изменение требует решения Архитектурного комитета + ИБ.

    [HC-01] Код НЕ покидает периметр банка.
            Все модели работают на инфраструктуре ГПБ (Феникс / Филин).
            Внешние LLM API запрещены.

    [HC-02] LLM НЕ имеет прямого доступа к репозиторию.
            В MVP: LLM живёт на клиенте (Kilo/Cline). MCP-сервер передаёт
            только структурированный JSON с найденными проблемами — не весь файл.
            В будущих фазах: LLM на сервере получает только сниппет до 40 строк.

    [HC-03] Автоматическое изменение кода запрещено.
            Любой патч — только предложение. Применение — только разработчиком.

    [HC-04] Human-in-the-loop обязателен.
            Разработчик подтверждает каждое изменение.

    [HC-05] Все действия логируются.
            Каждый вызов тула: метод, параметры, время, статус.
            В MVP: логи uvicorn. В будущем: структурированный JSONL.

    [HC-06] Изменение контракта между компонентами — только через RFC.

## 4. Как работает MVP прямо сейчас

    Разработчик → пишет в Kilo/Cline:
      "проверь этот Perl код" / "вызови perlcritic_analyze"
          ↓
    LLM на клиенте (Феникс/Qwen через Kilo) → решает вызвать тул
          ↓
    MCP-сервер (виртуалка · Docker · :8000)
      → запускает perlcritic внутри контейнера
      → парсит вывод в структурированный JSON
      → возвращает один структурированный текстовый блок
          ↓
    LLM на клиенте получает структурированный отчёт
    → объясняет разработчику: что не так, почему, как исправить
          ↓
    Разработчик исправляет код сам

**Ключевое:** В MVP LLM живёт на клиенте, не на сервере. MCP-сервер — умный исполнитель,
который возвращает максимально точные структурированные данные. Качество объяснений LLM
напрямую зависит от качества данных из тула.

### ВАЖНО: severity в Perl::Critic работает наоборот

    Severity 1 = САМЫЙ СТРОГИЙ (все нарушения, включая стилистические)
    Severity 2 = строгий
    Severity 3 = средний
    Severity 4 = мягкий (только серьёзное)
    Severity 5 = ТОЛЬКО КРИТИЧЕСКИЕ (минимальный фильтр)

    Это ПРОТИВОПОЛОЖНО интуиции. Все LLM часто путают это! Не путать.

    severity=1 → perlcritic --severity 1 → максимальная строгость, все нарушения
    severity=5 → perlcritic --severity 5 → только самые критические ошибки

    Pre-push hook: блокирует при severity >= 4 (только серьёзное и критическое)

### Передача кода в тул

MCP-сервер работает удалённо на виртуалке. Прямого доступа к файловой системе разработчика нет.

| Способ         | Когда работает                                     | Ограничения                            |
|----------------|----------------------------------------------------|----------------------------------------|
| code (строка)  | Всегда                                             | Нет доступа к другим файлам проекта    |
| target (путь)  | Только если путь доступен внутри контейнера        | В типичном деплое недоступно           |

**Рекомендуется для удалённого MCP:** передавать код строкой через параметр `code`.
Pre-push hook делает именно так — читает файлы локально и передаёт содержимое.

## 5. Контракт тула perlcritic_analyze (MVP)

### Входные параметры

    target     : string  — путь к файлу (только если доступен в контейнере)
    code       : string  — Perl-код строкой (рекомендуется для удалённого MCP)
    filename   : string  — имя файла для отчёта (опционально, default: analysis.pl)
    severity   : integer — 1-5 (1=строжайший все нарушения, 5=только критические, default: 1)
    recursive  : boolean — рекурсивный анализ директорий (default: true)
    statistics : boolean — включить статистику perlcritic (default: true)
    count_only : boolean — только количество проблем (default: false)

### IssueObject (v1.0 MVP)

    file     : string  — имя файла (basename)
    line     : integer — номер строки
    issue    : string  — человекочитаемое описание проблемы
    severity : integer — 1-5 (1=строжайший, 5=только критические)
    policy   : string  — полное имя политики: InputOutput::ProhibitTwoArgOpen
    snippet  : string  — строка кода с ошибкой (доступна когда код передан через code)

**Примечание по snippet:** Когда код передаётся через `code`, сниппет извлекается из
переданного кода по номеру строки. Когда через `target` — читается из файла в контейнере.
Если файл недоступен — snippet пустой. Это известное ограничение MVP.

**Примечание по policy и verbose-формату:** Используется кастомный формат запуска perlcritic
(--verbose "%p
%m at line %l, column %c.
(Severity: %s)
"), который даёт полное имя
политики в первой строке каждой ошибки — без конфликтов с regex-парсером.
Дефолтный формат perlcritic (без --verbose) даёт только ссылку на PBP (книгу Perl Best
Practices), но не имя policy. Формат --verbose 5 создавал проблемы: добавлял путь к файлу
в начало строки (/app/file.pl: ...) и regex не мог разобрать строки без пути.
Кастомный формат решает оба этих аспекта.

### Формат вывода для LLM (один структурированный блок)

Сырой вывод perlcritic LLM не передаётся — только нормализованные данные:

    PERL CODE ANALYSIS REPORT
    =========================
    File    : script.pl
    Severity: 1 (all issues — most strict)
    Total   : 3 issues found

    ISSUES
    ------

    [1/3] SEVERITY 5 — CRITICAL
    Policy : InputOutput::ProhibitTwoArgOpen
    Line   : 3, Column: 1
    Message: Two-argument "open"
    Snippet: open(FILE, $f);

    [2/3] SEVERITY 4 — HIGH
    Policy : InputOutput::ProhibitBarewordFileHandles
    Line   : 3, Column: 1
    Message: Bareword file handle opened
    Snippet: open(FILE, $f);

    SUMMARY
    -------
    Severity 1 (most strict / all) : 0
    Severity 2                     : 0
    Severity 3 (medium)            : 0
    Severity 4 (high)              : 1
    Severity 5 (critical only)     : 1

## 6. Ограничения MVP и почему они именно такие

| Ограничение                          | Причина                                    | Планируемое решение           |
|--------------------------------------|--------------------------------------------|-------------------------------|
| Snippet пустой при target удалённо   | Нет доступа к FS разработчика              | При code — работает уже сейчас |
| file = basename, не полный путь      | perlcritic дефолт не всегда пишет путь     | Фаза 2: кастомный verbose     |
| LLM не на сервере                    | MVP: LLM на клиенте                        | Фаза 2: Explainer на сервере  |
| Нет RAG / кэша объяснений            | MVP                                        | Фаза 3                        |
| Нет анализа зависимостей (PPI)       | MVP                                        | Следующий приоритет           |

## 7. Будущее видение — JARVIS (Фаза 2+)

> Это видение, не текущее состояние. Каждый шаг реализуется через RFC.

### Фазы развития

| Фаза         | Название    | Что добавляется                         | Ценность                          |
|--------------|-------------|-----------------------------------------|-----------------------------------|
| 1 · Сейчас   | Линтер      | perlcritic_analyze, pre-push hook       | Автоматическая проверка           |
| 2 · Следующая| Навигатор   | parse_structure (PPI), explain_module   | Понимание легаси                  |
| 3 · Потом    | Аналитик    | get_call_graph, find_usages, RAG-кэш    | Безопасный рефакторинг            |
| 4 · Амбиция  | Рефакторер  | suggest_refactor, Explainer на сервере  | Активное снижение техдолга        |

### Целевая архитектура (Фаза 2-3)

    Разработчик (Kilo / opencode)
        ↓
    MCP-сервер — JARVIS:
      ├── perlcritic_analyze   → что плохо в коде (уже есть)
      ├── parse_structure      → карта функций и зависимостей (PPI)
      ├── get_call_graph       → кто кого вызывает
      ├── find_usages          → где используется функция X
      ├── explain_module       → объяснение через Феникс (LLM на сервере)
      └── suggest_refactor     → безопасные шаги рефакторинга

    Феникс / Филин (Qwen 397B) — встраивается в сервер в Фазе 2+:
      → получает IssueObject (не сырой текст)
      → сниппет до 40 строк (не весь файл)
      → объясняет, предлагает патч
      → разработчик принимает решение

### Принцип нормализации (обязателен с Фазы 2)

Когда LLM переедет на сервер:

    Сырой текст perlcritic → LLM        ЗАПРЕЩЕНО
    JSON без snippet → LLM              допустимо только для классификации
    JSON + snippet кода → LLM           обязательный формат для объяснений

В MVP это правило выполняется: сырой вывод не передаётся LLM, только структурированный JSON.

### Полная карта тулов (целевая)

| Тул                | Что делает                              | Статус           | Технология        |
|--------------------|-----------------------------------------|------------------|-------------------|
| drospr             | Проверка соединения                     | ГОТОВ            | —                 |
| perlcritic_analyze | Анализ файла или code-строки            | ГОТОВ            | Perl::Critic      |
| parse_structure    | Карта функций, переменных, зависимостей | СЛЕДУЮЩИЙ        | PPI               |
| get_call_graph     | Граф вызовов                            | ПЛАНИРУЕТСЯ      | PPI + граф        |
| find_usages        | Где используется функция X              | ПЛАНИРУЕТСЯ      | PPI + grep        |
| explain_module     | Объяснение на русском                   | ПЛАНИРУЕТСЯ      | Феникс/Qwen       |
| suggest_refactor   | Предложение безопасных шагов            | ПОТОМ            | Феникс/Qwen       |

### IssueObject целевой (Фаза 2)

    file           : string  — относительный путь от корня репо
    line           : integer — номер строки
    col            : integer — колонка (опционально)
    issue          : string  — человекочитаемое описание
    policy         : string  — полное имя: InputOutput::ProhibitTwoArgOpen
    severity       : integer — 1-5 (1=строжайший, 5=только критические)
    snippet        : string  — строка с ошибкой (обязательно в Фазе 2)
    context_before : string  — до 3 строк до (опционально)
    context_after  : string  — до 3 строк после (опционально)
