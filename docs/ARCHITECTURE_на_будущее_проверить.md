## 1. Назначение и область действия

Этот документ — **единственный источник истины** об архитектурных решениях системы.  
Он обязателен для всех: разработчиков, DevOps, архитекторов, ИБ, аудиторов.

**Что система делает:**
- Автоматический статический анализ Perl-кода через `Perl::Critic`
- Группировка и приоритизация найденных проблем
- AI-объяснение каждой проблемы (что не так, почему, как исправить)
- Предложение готовых патчей (только предложение — не автоприменение)

**Что система НЕ делает:**
- Не изменяет код автоматически — никогда, ни при каких условиях
- Не отправляет код за пределы периметра банка
- Не имеет прямого доступа к репозиторию из LLM-компонентов
- Не принимает решений без подтверждения человека

## 2. Абсолютные ограничения (HARD CONSTRAINTS)

> Нарушение любого из этих пунктов — основание для немедленного останова системы.  
> Изменение этих пунктов требует решения Архитектурного комитета + ИБ + Регулятора.

```
[HC-01] Код НЕ покидает периметр банка.
        Все модели работают на инфраструктуре ГПБ (Феникс / Филин).
        Внешние LLM API запрещены.

[HC-02] LLM НЕ имеет прямого доступа к репозиторию.
        Код передаётся только через контролируемый pipeline:
        файл → Scanner → нормализованный JSON → Explainer → LLM.
        LLM никогда не читает файлы напрямую.

[HC-03] Автоматическое изменение кода запрещено.
        Любой патч — только предложение (diff).
        Применение — только через PR + code review + merge разработчиком.

[HC-04] Human-in-the-loop обязателен на каждом выходе.
        Разработчик подтверждает каждое изменение.
        Система не может "нажать применить" сама.

[HC-05] Все действия логируются.
        Каждый вызов LLM: агент, токены, модель, время, статус.
        Логи хранятся минимум 1 год. Доступ — только ИБ + аудит.

[HC-06] Изменение контракта между компонентами — только через RFC.
        RFC требует одобрения: владельца слоя + смежного слоя + архитектора.
```

## 3. Архитектурные принципы

### P1 — Нормализация до LLM (MANDATORY)

**Проблема:** `Perl::Critic` возвращает сырой текст. LLM плохо работает с неструктурированным вводом — теряет severity, путает строки, галлюцинирует детали.

**Решение:** Между Scanner и LLM всегда стоит слой нормализации.

```
Уровни риска:
  Сырой текст → LLM              🔴 ЗАПРЕЩЕНО
  JSON без сниппета → LLM        🟡 допустимо только для классификации
  JSON + snippet кода → LLM      🟢 обязательный формат для объяснений
```

**Обязательный формат перед LLM-вызовом:**

```json
{
  "file": "module/auth.pl",
  "line": 42,
  "issue": "Two-argument open",
  "severity": 4,
  "policy": "InputOutput::ProhibitTwoArgOpen",
  "snippet": "open(FILE, $path);",
  "context_before": "sub read_config {",
  "context_after": "    while (<FILE>) {"
}
```

**Правило:** Если объект не прошёл валидацию по схеме — он не уходит в LLM. Возвращается ошибка.

---

### P2 — Разделение слоёв (Separation of Concerns)

Каждый компонент отвечает за одно:

| Слой | Что делает | Что НЕ делает |
|------|-----------|---------------|
| Scanner | Запускает Perl::Critic, нормализует в JSON | Не интерпретирует, не вызывает LLM |
| Analyzer | Группирует, приоритизирует | Не читает файлы, не вызывает LLM |
| Explainer | Формирует промпт, вызывает LLM | Не запускает Perl::Critic |
| Refactor | Формирует diff-предложение | Не применяет патч сам |
| RAG | Кэш объяснений | Не принимает решений |
| LiteLLM | Маршрутизация + мониторинг | Не знает бизнес-логику |

---

### P3 — Fail-safe по умолчанию

```
Правило: при любой ошибке компонент возвращает {error: "...", result: null}.
         Он НЕ бросает исключение наверх.
         Он НЕ продолжает pipeline с некорректными данными.
         Он НЕ пытается "угадать" правильный результат.
```

---

### P4 — Версионирование контрактов и промптов

- Каждый контракт имеет версию: `v1.0`, `v1.1`, `v2.0`
- Каждый промпт хранится с версией: `prompts/explainer/v1_explain_issue.txt`
- В проде всегда работает явная версия — не `latest` в коде, а зафиксированная в конфиге
- Откат к предыдущей версии — за 5 минут (горячая замена конфига)

---

### P5 — Минимальный доступ к коду

```
LLM видит только:
  - фрагмент кода с проблемой (±20 строк)
  - тип проблемы (policy name)
  - severity

LLM НЕ видит:
  - весь файл целиком
  - другие файлы
  - историю git
  - имена разработчиков
  - бизнес-комментарии в коде (если в них PII или конфиденциальные данные)
```

## 5. Конвейер обработки данных

### Шаг 1 — Scanner

```
ВХОД:  путь к файлу или модулю (.pl / .pm)
ДЕЙСТВИЕ:
  1. Запустить Perl::Critic с заданным severity-порогом
  2. Распарсить вывод через regex (НЕ передавать сырой текст дальше)
  3. Для каждой проблемы сформировать IssueObject
  4. Читать сниппет: строка ошибки ±3 строки (через read_file tool)
  5. Валидировать каждый объект по JSON-схеме
  6. Вернуть массив валидных IssueObject + summary
ВЫХОД: { issues: IssueObject[], summary: { total: int, by_severity: {} }, error: string|null }

ПРАВИЛО: если файл не читается → error, issues = []. Не падать.
ПРАВИЛО: если Perl::Critic недоступен → error с диагнозом.
ПРАВИЛО: severity фильтр по умолчанию ≥ 3 (1-2 — шум для MVP).
```

### Шаг 2 — Analyzer

```
ВХОД:  IssueObject[] от Scanner
ДЕЙСТВИЕ:
  1. Сгруппировать по полю "policy" (тип проблемы)
  2. Посчитать частоту каждой группы
  3. Ранжировать: severity × log(count) — частые важные проблемы наверх
  4. Проверить RAG: есть ли уже объяснение для этого policy?
  5. Отдать топ-N групп (N настраивается, default = 10)
ВЫХОД: { groups: GroupObject[], rag_hits: string[], total_issues: int }

ПРАВИЛО: 200 одинаковых ошибок → 1 группа с count=200.
         Explainer вызывается 1 раз для группы, не для каждой строки.
```

### Шаг 3 — RAG lookup (внутри Analyzer)

```
ВХОД:  policy name (string)
ДЕЙСТВИЕ:
  1. Векторный поиск в Qdrant по policy name
  2. Если найдено с similarity > 0.92 → вернуть cached_explanation
  3. Если нет → вернуть null (идти к Explainer)
ВЫХОД: { explanation: string, fix_example: string } | null

ПРАВИЛО: RAG — кэш, не основной источник истины.
         Устаревшие объяснения (> 90 дней) помечаются как stale.
```

### Шаг 4 — Explainer (LLM-вызов)

```
ВХОД:  GroupObject + representative IssueObject (с snippet)
ДЕЙСТВИЕ:
  1. Прочитать фрагмент кода ±20 строк (read_file tool)
  2. Загрузить промпт-шаблон из prompts/explainer/v{N}.txt
  3. Собрать промпт: шаблон + JSON issue + сниппет
  4. Отправить в LiteLLM с metadata: { agent: "Explainer", policy: "...", file: "..." }
  5. Получить ответ, извлечь структурированные поля
  6. Сохранить объяснение в RAG (Qdrant upsert)
  7. Вернуть ExplanationObject
ВЫХОД: { explanation: string, fix_example: string, severity_reason: string,
          from_cache: bool, tokens_used: int, error: string|null }

ПРАВИЛО: LLM видит ТОЛЬКО сниппет (≤ 40 строк). Не весь файл.
ПРАВИЛО: промпт НЕ содержит путь к файлу полностью (только имя файла).
ПРАВИЛО: если LLM вернул ошибку → error, explanation = null. Не ретраить > 2 раз.
```

### Шаг 5 — Refactor (только предложение, MVP+)

```
ВХОД:  IssueObject + ExplanationObject + original_snippet
ДЕЙСТВИЕ:
  1. Сформировать промпт с задачей: "сгенерируй исправленный фрагмент"
  2. Вызвать LLM (отдельный промпт-шаблон prompts/refactor/v{N}.txt)
  3. Сформировать unified diff
  4. Вернуть патч разработчику в IDE
  5. ЖДАТЬ подтверждения — система переходит в состояние PENDING_APPROVAL
ВЫХОД: { patch: string (unified diff), explanation: string, requires_approval: true }

ПРАВИЛО: requires_approval всегда true. Без исключений.
ПРАВИЛО: система не может применить патч сама. Код применения — только в Kilo/Cline на стороне разработчика.
ПРАВИЛО: патч действителен 24 часа. После истечения — запрашивается заново.
```

## 6. Контракты компонентов

> Изменение контракта — только через RFC. Версия фиксируется в CONTRACTS.md.

### IssueObject (v1.0)

```typescript
interface IssueObject {
  file: string;          // относительный путь от корня репо
  line: number;          // номер строки (1-based)
  col?: number;          // колонка (опционально)
  issue: string;         // человекочитаемое описание
  policy: string;        // Perl::Critic policy name (напр. "InputOutput::ProhibitTwoArgOpen")
  severity: 1 | 2 | 3 | 4 | 5;  // строго 1-5
  snippet: string;       // строка с ошибкой (одна строка)
  context_before?: string;  // ≤3 строки до (опционально)
  context_after?: string;   // ≤3 строки после (опционально)
}
```

### GroupObject (v1.0)

```typescript
interface GroupObject {
  policy: string;           // Perl::Critic policy name
  count: number;            // сколько вхождений в файле/модуле
  severity: number;         // максимальный severity в группе
  priority_score: number;   // severity × log(count), для сортировки
  representative: IssueObject;  // один пример для Explainer
  all_lines: number[];      // все строки где встречается (для отчёта)
  rag_hit: boolean;         // есть ли кэш в RAG
}
```

### ExplanationObject (v1.0)

```typescript
interface ExplanationObject {
  policy: string;
  explanation: string;      // что не так и почему
  fix_example: string;      // пример исправленного кода (всегда валидный Perl)
  severity_reason: string;  // почему это важно для банковского кода
  from_cache: boolean;      // true = взято из RAG, LLM не вызывался
  tokens_used: number;      // 0 если from_cache=true
  model_used?: string;      // какая модель ответила
  error: string | null;     // null если успех
}
```

### PatchObject (v1.0)

```typescript
interface PatchObject {
  issue_ref: string;         // policy + file + line (идентификатор)
  patch: string;             // unified diff format
  original_snippet: string;  // что было
  fixed_snippet: string;     // что предлагается
  explanation: string;       // почему это исправление правильное
  requires_approval: true;   // ВСЕГДА true, не менять
  expires_at: string;        // ISO 8601, +24h от создания
}
```

## 8. Правила работы с LLM

### Обязательные правила (нарушение = блокировка PR)

```
[LLM-01] Сырой текст Perl::Critic НЕ уходит в LLM.
          Только нормализованный IssueObject + snippet.

[LLM-02] Размер контекста: snippet ≤ 40 строк.
          Если нужно больше — проблема декомпозирована неверно.

[LLM-03] Каждый LLM-вызов содержит metadata:
          { agent: string, policy: string, task: string }
          Это необходимо для аудита в LiteLLM.

[LLM-04] Промпт хранится в файле, не в коде.
          Запрещено: prompt = f"Объясни {issue}" прямо в коде.
          Правильно: prompt = registry.get("explainer", version="v1")

[LLM-05] Ответ LLM валидируется перед использованием.
          Если поля отсутствуют или тип неверный — error, не fallback.

[LLM-06] Retry policy: максимум 2 ретрая с exponential backoff.
          После 2 неудач — вернуть error пользователю.

[LLM-07] Timeout: 60 секунд на LLM-вызов.
          После — timeout error, не ждать бесконечно.
```

### Структура промпта (обязательный формат)

```
prompts/explainer/v1_explain_issue.txt:

---SYSTEM---
Ты — эксперт по Perl и безопасности банковского кода.
Отвечай только на русском языке.
Не добавляй информацию которой нет в предоставленных данных.
Формат ответа — строго JSON.

---USER---
Проблема в коде:
{{ISSUE_JSON}}

Фрагмент кода:
```perl
{{CODE_SNIPPET}}
```

Ответь в формате:
{
  "explanation": "что не так и почему это важно",
  "fix_example": "исправленный perl-код",
  "severity_reason": "почему severity={{SEVERITY}} для банковского кода"
}
---
```

## 9. Аудит и логирование

### Что логируется (обязательно)

```yaml
# Каждый LLM-вызов через LiteLLM логирует автоматически:
- timestamp
- agent_name          # из metadata.agent
- model_used          # qwen-397b / deepseek-3.2
- tokens_input
- tokens_output
- latency_ms
- status              # success / error / timeout
- policy              # из metadata.policy (тип проблемы)
- task                # из metadata.task

# Дополнительно MCP-сервер логирует:
- developer_id        # кто инициировал (из MCP-сессии)
- file_audited        # какой файл анализировался (только имя, не путь)
- issues_found        # сколько проблем найдено
- patches_proposed    # сколько патчей предложено
- patches_approved    # сколько патчей одобрено разработчиком
- patches_rejected    # сколько патчей отклонено
```

### Что НЕ логируется

```
НЕ логировать:
- полный код файла
- содержимое промптов с кодом (только policy name)
- персональные данные из комментариев в коде
- git-история
```

### Хранение логов

```
Место хранения:  виртуалка БСК, изолированный volume
Срок хранения:   минимум 1 год (требование регулятора)
Доступ:          ИБ + аудит (read-only)
Формат:          JSONL (одна запись = одна строка JSON)
Ротация:         ежедневно, сжатие gzip после 24 часов

## 10. RFC-процесс

> RFC (Request for Comments) — обязательная процедура для архитектурных изменений.

### Когда нужен RFC

- Добавить новый инструмент (тул)
- Изменить контракт между компонентами (поля IssueObject и др.)
- Изменить модель или провайдера LLM
- Добавить новый агент (например, включить Refactor)
- Изменить правила логирования или хранения данных
- Изменить порог схожести в RAG

### Шаблон RFC (создаётся как `rfc/RFC-NNN-название.md`)

```markdown
# RFC-NNN: Краткое название изменения

**Статус:** ПРЕДЛОЖЕНИЕ / ПРИНЯТО / ОТКЛОНЕНО  
**Автор:** имя  
**Дата:** ГГГГ-ММ-ДД  
**Слой:** tools / prompts / rag / infra / contracts / architecture

## Проблема
Что сейчас не работает или требует улучшения.

## Предлагаемое решение
Что конкретно меняется.

## Влияние на контракты
Какие JSON-схемы меняются (до/после).

## Кто затронут
Какие роли/компоненты должны адаптироваться.

## Риски
Что может пойти не так.

## Критерии принятия
Как поймём что изменение работает.

## Одобрения
- [ ] Владелец слоя: ___
- [ ] Смежный слой: ___
- [ ] Архитектор: ___
- [ ] ИБ (если нужно): ___
```

### Сроки RFC

```
Простые изменения (один слой):   1 рабочий день на сбор одобрений
Контрактные изменения:            3 рабочих дня
Изменения с участием ИБ:          по регламенту ИБ (обычно 5 дней)
Изменения HC-* ограничений:       запрещены без Архитектурного комитета
```

### Правила создания тасков в Speckit

```
[SPEC-01] Каждый таск ссылается на RFC или архитектурный принцип.
           Формат: "Реализует RFC-042" или "Следует принципу P1"

[SPEC-02] Таск на новый тул ОБЯЗАН содержать:
           - ссылку на контракт в CONTRACTS.md
           - acceptance criteria: "тест на контракт зелёный"
           - указание слоя: tools / prompts / rag / infra

[SPEC-03] Таск НЕ закрывается без зелёного теста на контракт.

[SPEC-04] Definition of Done для любого таска:
           ✅ код покрыт тестом на контракт
           ✅ PR затрагивает только свой слой
           ✅ CONTRACTS.md обновлён (если контракт менялся)
           ✅ ARCHITECTURE.md не противоречит изменению
           ✅ лог-запись LiteLLM содержит правильные metadata
```

### Шаблон spec-файла для агента

## Обязательное поведение

- PPI ≥ 1.276

## Метрики успеха

- Ошибки парсинга: 0%
